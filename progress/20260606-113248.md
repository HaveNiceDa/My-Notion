# 流式续跑可用性收口

## 目标

- 将已落地的 checkpoint/resume 后端能力暴露为用户可触发的“继续生成”入口。
- 在 checkpoint 恢复时重建完整 `currentDocument` 上下文，恢复依赖当前文档的 Agent tools。

## 已完成

- `useAIChatStream` 新增 `resumeLastRun` 和 `canResumeLastRun`，从 `sessionStorage` 读取 `AgentStreamResumeCursor` 并发起 resume 请求。
- `MessageList` 在最后一条 assistant 消息下展示“继续生成”按钮，点击后继续同一条消息的流式输出。
- 中断后保留已有部分输出，不再用通用错误文案覆盖已生成内容。
- 完成后清理 resume cursor，避免已完成 run 误显示“继续生成”。
- `/api/agent/stream` 在 `resumeFromCheckpoint` 中根据 checkpoint 的 `currentDocument.id` 查询最新文档，恢复完整 `currentDocument` 到 `buildAvailableTools` 和 `toolContext`。
- 更新 `docs/agent-stream-resume-protocol.md`，将 UI 入口和完整当前文档恢复标记为 Phase 3 已完成内容。

## 已知缺口

- running run 的 live stream 接管仍未实现；当前“继续生成”适用于已有 cursor 的 backlog replay、completed/failed run 处理和 failed checkpoint 恢复。
- 如果 checkpoint 中的 `currentDocument.id` 已失效或用户无权访问，恢复会降级为无当前文档上下文继续执行。
- 已保存的历史 assistant 消息暂不做原地更新；如果中断消息已落库，resume 完成后不会覆盖旧 Convex 消息。

## 下一步

- 为 running run 设计 live stream 接管策略，避免仅 replay backlog。
- 将 `agentRunEvents` 与 trace sink 关联，进入 Trace Replay UI。
- 如需更强一致性，补充 assistant 消息原地更新 mutation。
