# Deep Retrieval Query Rewrite

## 本次目标

进入 Hybrid Retrieval 的 `deep` 策略，实现 query rewrite：生成原问题、关键词版、语义扩展版，并接入 multi-query 混合检索。

## 关键决策

- `deep` 策略优先让独立 LLM 生成 query rewrite，而不是把语义扩展规则硬编码在业务逻辑里。
- 使用已有 OpenAI Compatible / DashScope 配置，默认模型来自 `DEFAULT_MODEL`，可通过 `RETRIEVAL_REWRITE_MODEL` 覆盖。
- LLM 未配置或调用失败时，回退到本地确定性改写，保证检索链路可用。

## 已完成

- 新增 `packages/ai/server/retrieval/query-rewrite.ts`：
  - 输出 `original`、`keyword`、`semantic` 三类 query variant。
  - LLM prompt 要求只返回 JSON：`keywordQuery` 和 `semanticQuery`。
  - 支持解析带 Markdown/额外文本包裹的 JSON。
  - 支持本地 fallback，避免缺少 `LLM_API_KEY` 时 deep 策略不可用。
- 改造 `retrieveKnowledge({ strategy: "deep" })`：
  - 不再复用 `balanced`。
  - 先执行 query rewrite。
  - 对每个 query variant 分别执行 `semanticRecall`、`keywordRecall`、`metadataRecall`。
  - 将所有候选统一交给 RRF fusion。
  - 在结果 metadata 中返回 `queryVariants`，便于调试和后续 trace。
- 扩展 retrieval 类型：
  - `QueryRewriteVariantKind`
  - `QueryRewriteVariant`
  - `KnowledgeRetrievalResult.metadata.queryVariants`
- 补充 focused tests：
  - `query-rewrite.test.ts` 覆盖 LLM rewrite 和 fallback。
  - `retrieval.test.ts` 覆盖 deep multi-query 三路召回。

## 已知风险

- `deep` 当前会对三个 query variant 都执行三路召回，成本和延迟高于 `balanced`，不适合作为默认策略。
- query rewrite 使用通用聊天模型，后续可以替换为更便宜的轻量模型或专门的 rewrite prompt cache。
- 当前还没有 reranker，multi-query 扩召后仍依赖 RRF 排序；复杂问题后续需要增加 rerank/context packing。

## 下一步建议

1. 为 `deep` 增加 retrieval trace，记录 queryVariants、每路召回数量、融合前后排名，给后续 harness/eval 使用。
2. 增加 rerank 层，先做轻量规则 rerank，再评估是否引入 LLM/reranker 模型。
3. 做 context packing，把同文档相邻 chunk 聚合成更适合回答的上下文窗口。
