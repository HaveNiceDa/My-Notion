# Retrieval Eval 工程化

## 本轮目标

- 将 retrieval eval 从单个测试文件升级为可持续运行的质量基线。
- 抽出 golden set fixture，补聚合指标和 CLI runner。
- 新增 `pnpm eval:retrieval`，为后续 CI smoke 和真实数据 eval 做准备。

## 已完成变更

- `packages/ai/eval/retrieval-eval-fixtures.ts`：
  - 抽出 `RetrievalEvalDoc`、`RetrievalEvalCase` 和 `RETRIEVAL_EVAL_CASES`。
  - 保留 3 个 synthetic golden set：`hybrid-api-contract`、`metadata-recency`、`token-budget-truncation`。
- `packages/ai/eval/retrieval-evaluator.ts`：
  - 新增 `runSyntheticRetrieval`，用固定分数模拟 semantic / keyword / metadata 三路召回。
  - 复用正式链路中的 `fuseCandidates`、`packRetrievalContext`、`buildCitationQuality`。
  - 新增 `runRetrievalEval`，输出聚合指标。
  - 新增 `formatRetrievalEvalSummary`，输出稳定 console summary。
- `packages/ai/eval/retrieval-eval-runner.ts`：
  - 新增 CLI runner。
  - eval 失败时设置非 0 `process.exitCode`，便于后续接 CI。
- `packages/ai/__tests__/retrieval-eval.test.ts`：
  - 改为复用 fixture 和 evaluator。
  - 测试聚合指标和单 case 通过状态，避免测试和 CLI 各维护一套断言。
- `package.json`：
  - 新增 `pnpm eval:retrieval`。
- `docs/ai-chat-refactor-plan.md`：
  - 更新 Harness / Retrieval Eval 状态，记录 fixture、聚合指标和 `pnpm eval:retrieval` 已完成。

## 已知风险

- 当前 runner 仍是 synthetic corpus，主要验证算法链路和质量信号，不代表真实数据分布。
- `recall@k` 和 MRR 当前基于期望 `documentId`，还没有细化到 chunk 级别。
- 尚未接入 CI smoke。
- `deep` 策略、Query Rewrite、多 query 的 eval 仍需单独补充。

## 下一步建议

1. 接入 CI smoke，先跑 `pnpm eval:retrieval`。
2. 增加真实数据 fixture，覆盖真实文档标题、路径、标签、最近编辑等 metadata。
3. 增加 chunk 级期望、`recall@chunkK` 和 citation quality 阈值。
4. 再根据真实数据指标决定是否引入二阶段 rerank。
