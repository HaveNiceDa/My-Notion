# RAG Tool 直接执行与输入框开关评估

## 完成的改动

- 进一步修复 `RAG + 深度思考` 场景下的 DashScope `tool_choice` 兼容问题。
- 将显式 RAG 模式从“让模型首轮强制生成 `knowledge_search` tool call”改为“后端直接执行 `knowledge_search` tool”。
- `/api/agent/stream` 当前不再向 DashScope 发送 `tool_choice` 参数，彻底绕开 thinking mode 下 object/required tool choice 的限制。
- 保持第二轮答案生成继续使用用户选择的 thinking mode。
- 评估输入框 RAG 知识库 icon：它与当前问题有关，因为显式 RAG 开关会进入知识库检索分支；但不建议本次直接删除，需等 Agent 自动判断工具调用能力完成后再移除。

## 验证命令和结果

- `pnpm --filter @notion/web exec tsc --noEmit`: ✅
- `pnpm --filter @notion/web build`: ✅ 通过，有 Sentry auth token 和 edge runtime 常规 warning
- `pnpm --filter @notion/web lint`: ✅ 通过，有 4 个既有 warning
- `GetDiagnostics`: ✅ `/api/agent/stream/route.ts` 无错误，仅有拼写提示

## 已知缺口或风险

- 输入框仍保留 RAG 知识库 icon，作为 Agent 自动 tool 判断完成前的临时显式入口。
- 后端当前 `chat` 模式不会自动调用 `knowledge_search`，因此现在删除 RAG icon 会导致用户无法主动触发知识库检索。
- 后续如果实现通用 Agent loop，需要避免在 thinking mode 下使用 object/required 形式的 `tool_choice`。

## 下一步建议

- 新增 Agent `auto` 模式，让后端根据用户问题和当前文档上下文判断是否调用 `knowledge_search`。
- `auto` 模式稳定后，删除输入框 RAG 知识库 icon，并将 `mode` 从前端状态中移除。
- 接入 `document_read` tool 后，把“总结/翻译/深度剖析此页面”纳入同一套 Agent 判断。
