# PAT 设置页 500 修复与默认 Token 简化评估

## 背景

- 线上设置页打开 API tokens 区域时，`GET /api/cli/tokens` 返回 500。
- 用户反馈当前 PAT 流程过重：需要手动创建、只展示一次、关闭后无法再次复制；期望像密码设置一样默认一个 token，可隐藏/显示明文，方便复制给 CLI 使用。

## 本次修复

- `apps/web/src/app/api/cli/tokens/route.ts`
  - `GET /api/cli/tokens` 查询失败时不再直接返回 500。
  - 对 token 列表查询做降级响应：返回 `success: true`、空列表、`unavailable: true` 和 warning，设置页可继续打开。
  - `POST` / `DELETE` 保留失败语义，但增加 `UNAUTHORIZED` / `SERVICE_UNAVAILABLE` / `INTERNAL_ERROR` 结构化错误。
- `apps/web/src/components/modals/settings-modal.tsx`
  - 支持读取 GET 降级响应里的 `warning`，在设置页内展示错误信息，而不是只在浏览器控制台暴露 500。
- `apps/web/convex/cli.ts`
  - 从 `export *` 改为显式导出 CLI/Machine API 相关 Convex functions，降低线上 codegen/deploy 对 re-export 感知不一致的风险。

## 验证

- `pnpm --filter @notion/web exec convex codegen`：通过。
- `pnpm --filter @notion/web typecheck`：通过。
- `pnpm --filter @notion/web lint`：通过，0 error / 7 warnings；warning 为既有图片优化和 hook deps 提示。

## 简化方案评估

### 当前设计

- Web 登录态创建 PAT。
- 明文 PAT 只在创建时返回一次。
- Convex `apiTokens` 表只存 `tokenHash`，不存明文。
- CLI 只消费 PAT，不负责创建 PAT。

### 用户期望

- 每个登录用户默认有一个 CLI token。
- 设置页打开后能直接看到已有 token。
- token 像密码框一样可隐藏/显示明文、复制。
- 不需要创建多个 token、scope、过期时间等复杂配置。

### 改造影响

- CLI 本身几乎不用改，仍然 `auth login --token mnt_xxx`。
- Web 设置页需要从“多 token 管理”改成“默认 token 管理”。
- 后端需要改变安全模型：如果要再次展示明文，就必须存明文或可逆密文；当前只存 hash，无法反推出历史 token。
- 旧 token 只能继续展示 prefix；要显示完整明文，需要重置/生成新的默认 token。

### 推荐方案

- 保留 hash 校验，新增 `tokenPlaintext` 或后续改为加密密文字段，仅用于设置页展示/复制。
- 每个用户维护一个默认 token：`name = "Default CLI Token"`，scope 固定为 `docs:read` + `docs:write`，默认不过期。
- GET 设置页时执行 ensure：没有默认 token 就自动创建；已有但没有明文字段则提示“旧 token 不可查看，请重置生成可复制 token”。
- UI 改成单卡片：密码框、显示/隐藏、复制、重置 token、撤销/禁用。
- CLI 文档改成：打开设置 -> 复制默认 token -> `my-notion auth login --token <token>`。

## 下一步建议

1. 先部署本次 500 修复，确认线上设置页不再报 GET 500。
2. 如接受“服务端可再次展示 token 明文”的便利性取舍，再做默认 token 改造。
3. 默认 token 改造完成后，更新 CLI quick start、README、Skills 文档和 E2E。
