# RAG Context Packing

## 本轮目标

- 新建 `context-packing.ts`，在 Hybrid Retrieval 的 fusion/dedup 后补上下文打包层。
- 按文档分组合并相邻 chunk，减少同一文档碎片化引用。
- 增加 token budget 裁剪，控制传给 Agent 的上下文长度。

## 已完成变更

- `packages/ai/server/retrieval/context-packing.ts`：
  - 新增 `packRetrievalContext(options)`。
  - 按 `documentId` 分组，并按 `chunkIndex` 排序。
  - 仅合并同一文档中 `chunkIndex` 连续的 chunk。
  - 合并后保留来源集合、最高融合分数、`packedChunkIds`、`packedChunkIndexes`、`packedItemCount` 和 `estimatedTokens`。
  - 支持默认 2400 token budget，也支持通过 `contextTokenBudget` 覆盖。
  - 超预算时按优先级保留高分 packed item，必要时截断最后一个 item，并标记 `truncatedByTokenBudget`。
- `packages/ai/server/retrieval/types.ts`：
  - `KnowledgeRetrievalOptions` 新增 `contextTokenBudget`。
  - `KnowledgeRetrievalResult.metadata` 新增 `packedCount`、`contextTokenBudget`、`contextEstimatedTokens`、`contextTruncated`。
- `packages/ai/server/retrieval/index.ts`：
  - `fast` 策略保持原低延迟语义召回行为，不做 packing。
  - `balanced` 和 `deep` 在 RRF fusion 后执行 Context Packing。
  - `fusedCount` 保留 fusion 后、packing 前的候选数；`packedCount` 表示最终返回上下文数。

  - 新增 token budget 裁剪测试。

  - 后续 RAG 质量重点调整为 Citation Quality 和必要时二阶段 rerank。

## 设计说明

- Packing 放在 fusion 后：先让 semantic / keyword / metadata 多路召回和 RRF 决定候选优先级，再合并相邻上下文。
- 只合并相邻 chunk：避免把同一文档内距离较远的段落硬拼在一起，降低上下文污染。
- token 估算采用轻量近似：中文字符按 1 token，非中文字符按约 4 字符 1 token，用于预算裁剪而非计费。
- `fast` 不接 packing：保持实时聊天低延迟路径尽量简单。

## 已知风险

- 当前 token budget 是近似估算，不等同于模型 tokenizer 的精确 token 数。
- Packing 只基于已召回并融合后的候选，不会额外回表拉取未召回的相邻 chunk。
- Citation Quality 仍未完成，引用覆盖率、命中分数解释和是否需要继续检索等质量信号仍待补。

## 下一步建议

1. 做 Citation Quality：输出引用覆盖率、来源分数解释、packing/truncation 说明。
2. 在 Web Agent `knowledge_search` tool 卡片中展示 `packedCount`、`contextTruncated` 等 RAG 质量信号。
3. 视召回 eval 结果决定是否接二阶段 rerank。
