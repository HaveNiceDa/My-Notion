# Phase 5 P0：requestId 全链路透传

## 目标

完成机器 API 安全与可观测性的第一步：为 `/cli/v1/*` 响应增加稳定 `requestId`，并让 CLI/MCP 错误链路透出该 id，便于排查 E2E 或线上失败。

## 已完成

- `apps/web/convex/http.ts`：
  - 每个 CLI HTTP Action 请求生成 `req_` 前缀 requestId。
  - 成功响应增加 JSON 字段 `requestId`。
  - 错误响应增加 JSON 字段 `requestId`。
  - 所有响应增加 `x-request-id` header。
  - 服务端 catch 日志增加 `requestId=...`。
- `packages/my-notion-cli`：
  - `ApiResponse` 类型增加可选 `requestId`。
  - `MyNotionApiError` 增加 `requestId` 字段。
  - CLI 错误消息自动附带 `(requestId: req_xxx)`。
- `packages/my-notion-skills`：
  - `references/cli-commands.md` 增加 Error Output 章节。
  - 已运行 `pnpm sync:skills` 同步 `.trae/skills`。
- 架构文档：
  - Phase 5 P0 中“CLI/MCP 错误输出透出 requestId”已标记完成。

## 验证结果

- `pnpm --filter @notion/web exec convex codegen`：通过。
- `pnpm --filter @notion/my-notion-cli typecheck`：通过。
- `pnpm --filter @notion/my-notion-cli build`：通过。
- `pnpm --filter @notion/web exec tsc --noEmit`：通过。
- 无效 token 直连 HTTP 验证：
  - 响应 header 包含 `x-request-id`。
  - 响应 body 包含 `requestId`。
- 无效 token CLI 验证：
  - 输出 `UNAUTHORIZED (requestId: req_xxx)`。
- `pnpm e2e:cli`：
  - 第一次运行在 `docs create` 遇到网络层 `fetch failed`，无服务端响应，因此无 requestId。
  - 第二次运行在 `docs archive` 遇到网络层 `fetch failed`，finally 仍执行兜底归档和 PAT cleanup。
  - 第三次复跑通过。
- `pnpm e2e:mcp`：通过。
- `GetDiagnostics`：无诊断错误。

## 当前结论

- requestId 已覆盖服务端结构化响应和 CLI/MCP HTTP API 错误链路。
- 网络层 `fetch failed` 不会携带 requestId，因为请求未拿到服务端响应；这类问题后续需要通过重试、超时策略或网络诊断继续治理。
