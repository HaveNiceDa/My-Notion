# Agent Tool 目录拆分

## 完成的改动

- 新增 `apps/web/src/lib/agent/tools/` 目录，沉淀 Agent tool 相关能力。
- 新增 `types.ts`，集中定义 `CurrentDocumentContext`、`PendingToolCall`、`ToolExecutionResult`。
- 新增 `document-read.ts`，包含 `document_read` 的触发判断、tool call 创建和执行逻辑。
- 新增 `knowledge-search.ts`，包含 `knowledge_search` 的触发判断、tool call 创建和执行逻辑。
- 新增 `index.ts`，统一导出 tools 类型和能力函数。
- 精简 `/api/agent/stream/route.ts`，移除内联 tool 函数，只保留 Agent 编排、OpenAI stream 和 NDJSON 事件协议。
- 保持现有行为不变：仍由服务端直接执行 tool，不向 DashScope 发送 object/required `tool_choice`。

## 验证命令和结果

- `pnpm --filter @notion/web exec tsc --noEmit`: ✅
- `pnpm --filter @notion/web lint`: ✅ 通过，有 4 个既有 warning
- `pnpm --filter @notion/web build`: ✅ 通过，有 Sentry auth token 和 edge runtime 常规 warning
- `GetDiagnostics`: ✅ 新增 tool 文件无错误；Agent route 仅有拼写提示

## 已知缺口或风险

- Tool planner 仍在 route 中以 `shouldReadDocument / shouldSearch` 组合表达，后续可继续拆到 `lib/agent/planner/`。
- `document_read` 仍使用客户端传入的当前文档上下文，服务端 Convex auth 读取尚未实现。
- `ToolExecutionResult` 暂未被使用，保留给后续统一 tool executor 返回结构。

## 下一步建议

- 拆分 `lib/agent/planner`，统一管理 tool 选择策略。
- 引入通用 `executeTool(toolCall, context)`，减少 route 中的 if/else。
- Agent 主路径稳定后清理旧 RAG API。
