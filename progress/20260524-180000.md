# 3.4 API Rate Limiting

## 完成的改动

- 新增 `lib/agent/rate-limiter.ts`：纯内存滑动窗口限流（20 次/分钟/用户）
  - 零外部依赖，无需 Upstash Redis 或环境变量
  - 每 5 分钟自动清理过期记录，防止内存泄漏
  - 进程重启后计数重置（个人项目可接受）
- 修改 `app/api/agent/stream/route.ts`：在 Clerk auth 校验后、请求体解析前插入限流检查
  - 429 响应包含 `X-RateLimit-Limit`、`X-RateLimit-Remaining`、`X-RateLimit-Reset`、`Retry-After` 头
- 修改 `components/ai-chat/stream-client.ts`：前端 429 响应处理，显示中文友好提示（含等待秒数）

## 验证命令和结果

- `pnpm exec tsc --noEmit`: ✅
- `pnpm build`: ✅

## 已知限制

- 纯内存方案：进程重启后计数重置，多实例部署时限流不共享
- 对个人项目（单实例 Vercel）足够使用

## 下一步建议

- 如需多实例共享限流，可切换为 Upstash Redis 方案
- 考虑对不同模型设置不同限流阈值
