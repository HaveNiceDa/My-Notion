重新启动此前已阶段性收口的 Agent 重构路线，把下一阶段重点从“Agent 能跑”调整为“工具更丰富、记忆更稳定、检索更可靠”。

## 已完成

- 梳理现有里程碑状态：M10-M14 已完成 Web AI Chat / ReAct Agent Loop，M16 已完成 CLI / Skills / MCP Agent 写文档链路。
- 确认当前 Agent 基线：已有 `knowledge_search`、`web_search`、`document_read`，但缺少长期记忆和更丰富的工具生态。
- 确认当前 RAG 缺口：现有检索主要是 Qdrant 向量检索、阈值过滤和 documentId 去重，尚未形成真正的 query rewrite、多路召回、重排和上下文组装。

## 已知风险

- Memory 设计涉及隐私、可解释性、删除权和冲突处理，不能只把数据写进向量数据库。
- Document write 类 tool 涉及用户数据修改，必须默认 dry-run 或 confirmationRequired。
- RAG 的 `deep` 策略可能显著增加延迟和 token 成本，需要保留 `fast/balanced/deep` 三档策略。

## 下一步建议

1. 先实现 `retrieveKnowledge(options)`，把 `knowledge_search` 背后的检索逻辑从单一 `similaritySearch` 抽象出来。
2. 再实现 `agentMemories` 数据模型和 `memory_read` / `memory_write` tool，确保写入可确认、可审计、可删除。
3. 随后实现 `document_write` dry-run tool，打通 Agent 修改文档的安全闭环。
4. Harness 暂时后置到 M18，但从 M17 开始记录 tool trace，避免后续无法回放。
