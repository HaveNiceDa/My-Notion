# ReAct 工具上限与重复 Tool 展示优化

## 本轮目标

- 修复 Agent 达到 `MAX_ITERATIONS` 后仍停留在工具调用阶段，导致最终无文本输出的问题。
- 优化重复调用同一 tool 时的 UI 展示，减少冗余卡片。

## 根因

- ReAct loop 在第 5 轮仍收到 `tool_calls` 时，会执行 tool 并结束 `for` 循环，随后直接发送 `finish`，没有再让 LLM 基于最后一轮工具结果生成最终回答。
- 前端按 tool call id 逐条展示卡片，同一轮回答里重复的只读 tool 调用会全部展开，视觉上冗余。

## 已完成变更

- ReAct loop 修复：
  - 达到工具迭代上限且最后一轮仍有 tool 调用时，追加一条 system 指令，强制进入最终回答阶段。
  - 最终回答阶段不再传 `tools` / `tool_choice`，避免模型继续调用工具。
  - 新增本轮 tool result cache：同名同参数 tool 在同一次 Agent run 内只真实执行一次，后续重复调用复用缓存结果。
- UI 展示优化：
  - `ToolCallResult` 记录 tool arguments，便于前端判断重复调用。
  - `MessageList` 对只读 tool 按 `toolName + arguments` 折叠重复卡片，并显示重复次数。
  - 写类 tool 不折叠，避免确认按钮和状态混淆。
  - `ToolCallCard` 补充 `web_extract` / `document_search` 的展示名称、图标和结果详情。
- i18n：
  - 补充 `webExtractTool`、`documentSearchTool`、`toolRepeated`、`webExtractTruncated` 的中英文文案。
- 测试：
  - 新增 `react-loop.test.ts`，覆盖“连续 5 轮重复 tool 调用后强制最终回答”和“重复 tool 只执行一次”。

## 已知风险

- 重复 tool cache 只在单次 Agent run 内生效，跨会话/跨请求缓存仍留给后续 `Tool 结果缓存` 任务。
- UI 折叠只针对只读 tool，写类 tool 保持逐条展示。
- 如果模型在最终无工具阶段仍无法生成内容，仍会返回空文本；但该阶段已不提供 tools，复现概率应显著降低。

## 下一步建议

1. 用 `https://w3c.github.io/touch-events/#mouse-events 总结这个链接` 复测，确认工具上限后会输出最终总结。
2. 观察重复 `web_extract` 是否折叠为一张卡片并显示重复次数。
3. 后续继续做跨请求 `Tool 结果缓存`，减少重复抓取和搜索成本。
