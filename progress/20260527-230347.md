# Memory 自动注入与语义检索

## 本轮目标

- 修复运行时 `Navigation.memories` 在 `en` locale 下偶发缺失导致的页面错误。
- 收口 Memory 剩余方向：
  - Agent Loop 自动注入相关长期记忆。
  - Memory 接入 embedding / Qdrant 语义检索，并提供降级策略。

## 已完成变更

- 修复 i18n 运行时缺 key 风险：
  - `Navigation.memories` 已存在于 `en.json` / `zh-CN.json`。
  - `Navigation` 增加 `t.has("memories")` fallback，避免 Turbopack / dev server 消息缓存导致单个 key 缺失时打爆页面。
- 新增 Memory 语义检索服务：
  - `packages/ai/server/memory.ts`
  - `retrieveRelevantMemories` 会优先同步 active memory 到 Qdrant，再用 embedding 做语义召回。
  - Qdrant 不可用或语义无命中时降级到轻量 token / recency 匹配。
- 扩展 Qdrant wrapper：
  - 支持 `upsertAgentMemory`。
  - 支持 `semanticSearchAgentMemories`。
  - 为 memory point 增加 `metadata.kind = "agentMemory"`、`memoryId`、`memoryType`、`confidence`、`source` 等元数据。
- Agent Stream 自动注入 memory：
  - 在 `/api/agent/stream` 中提取最新用户消息。
  - 请求开始前读取 active memory，检索相关记忆并写入 system prompt。
  - 当前轮明确用户指令优先级高于长期记忆。
- `memory_read` tool 升级：
  - 查询时先读取 active memory，再走 `retrieveRelevantMemories`。
  - tool result metadata 会返回 `retrieval`、`unavailable`、`error` 等信号。
- 补充单测 mock：
  - `tools.test.ts` mock `retrieveRelevantMemories`，避免单测访问真实 Qdrant。

## 已知风险

- Memory embedding 当前采用懒同步：Agent 请求或 `memory_read` 时才同步 active memory 到 Qdrant。
- 已停用 memory 的旧 Qdrant point 依赖 active memory id 过滤规避，不会被召回，但未做即时物理删除。
- 若 dev server 仍复现 `MISSING_MESSAGE`，需要重启 dev server 清掉 Turbopack 旧消息缓存。

## 下一步建议

1. 将 Memory 的 Qdrant 同步从懒同步升级为写入/编辑/停用时的显式同步任务。
2. 在 Memory Review UI 展示每条 memory 的 embedding 同步状态。
3. 给 Agent Memory 自动注入补专门单测或 E2E smoke。
